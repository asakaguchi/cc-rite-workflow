### Fix Patterns
- **末尾 `&& ` 短絡が script 最終文になると exit code を leak する (bash gotcha)**: error-surfacing 用に追加した cleanup 行 `[ -n "$mv_err" ] && rm -f "$mv_err"` が CLEANED 経路の最終実行文になり、`mv_err` 空時に `[ -n "" ]`=false → rc=1 を script exit code に leak。修正は **末尾 `exit 0` を明示**するのが最小・最低リスク (sibling helper 全てが従う規約)。`if [ -n "$mv_err" ]; then rm; fi` 形式でも解決可能だが、末尾 `exit 0` の方が将来の行追加に対して頑健。
- **非ブロッキング契約 (exit 0 always) は最終文の rc に依存させない**: `set -uo pipefail` のみで `-e` なしの helper は末尾 `exit 0` がないと最終コマンドの rc がそのまま script exit code になる。「Exit codes: 0 always」を文書化する helper は末尾 `exit 0` を明示すべき。
- **edge-case test の pin 方法 (mktemp shim の引数分岐)**: CLEANED 経路 + 内側 no-arg mktemp 失敗を再現するには、PATH-shim mktemp で `[ $# -eq 0 ]` 時のみ exit 1、テンプレート付き呼び出し (atomic temp) は real mktemp へ delegate する。S-5/S-6 の mv-shim 技法の拡張。
- **複数 reviewer の評価が割れたときの解決**: code-quality/security は「exit 0 維持」と評価したが、これは彼らがテストした path (normal CLEANED / mv-failure) では正しく、error-handling が指摘した特定 edge (CLEANED + 内側 mktemp 失敗) が未カバーだっただけ。真の矛盾ではなく coverage gap。より具体的な runtime evidence を持つ finding を実機再現で確証して採否を決めるのが正しい。
