### Fix Patterns
新規ヘルパースクリプトが失敗契約（exit code 検査必須）を文書化していても、
それを使う呼び出し元を機械的に複数箇所へ展開すると、Issue の Before/After 仕様例
（`2>/dev/null` パターン）をそのまま踏襲してしまい、契約を満たさない呼び出しが
量産されるリスクがある。レビュー時のクロスバリデーションでレビュアー間の重要度
判定が対立（CRITICAL vs 非ブロッキング）した場合、討論プロトコルの CRITICAL guard
により討論をスキップして即座にユーザーへエスカレーションし、安全側の判断を
仰ぐのが有効だった。修正パターンとしては、単一行の `cmd || fallback="非空文字列"`
（コマンド置換の exit code を直接利用）が既存の grep 抽出テストとの互換性を保ちながら
fail-safe を実現する上で有効。パイプを含む場合は PIPESTATUS がサブシェル境界を
越えないため使えず、代わりにコマンド置換の exit code を直接 `||` でチェックしてから
パラメータ展開で先頭行を取り出す方式が必要。
