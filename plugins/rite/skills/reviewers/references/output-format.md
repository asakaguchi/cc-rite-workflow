# Output Format Reference

Review result output format definitions.

## Individual Reports

Structured reports generated by each reviewer:

```markdown
### {Reviewer Type} Review

#### Domain-Specific Analysis
[Sections specific to each reviewer's expertise, e.g.:]
  - Security: Security Assessment
  - API: Endpoint Analysis, Contract Changes
  - Database: Schema Analysis, Query Performance, Migration Safety
  - Test: Coverage Analysis, Test Quality
  - DevOps: Pipeline Analysis, Container Analysis
  - Frontend: Component Analysis, Accessibility Audit, Performance Concerns
  - Dependencies: Security Scan, License Analysis, Bundle Impact
  - Prompt Engineer: Executability Analysis, Instruction Clarity
  - Tech Writer: Content Analysis, Link Verification

#### Findings
[Structured findings table - common to all reviewers]
| Severity | Scope | File:Line | Issue | Recommendation |
|----------|-------|-----------|-------|----------------|
| {level}  | {scope} | {location}| {WHAT + WHY} | {FIX + EXAMPLE} |

#### Summary/Recommendations
[Domain-specific recommendations and summary]
```

**Note**: Each skill file has domain-specific analysis sections, but the **Findings table** format is common to all reviewers.

**Column structure:** Scope column accepts `current-pr` / `follow-up` / `nit-noted` (schema 1.1.0+, Issue #1016) — see [_reviewer-base.md Scope Assignment Flowchart](../../../agents/_reviewer-base.md#scope-assignment-flowchart) and the [Severity × Scope Matrix](../../../references/severity-levels.md#severity--scope-matrix) for assignment rules and forbidden combinations. Issue column must include WHAT (what is wrong) + WHY (why it matters). Recommendation column must include FIX (how to fix) + EXAMPLE (code example when applicable).

## Unified Report

Unified report generated by the coordinator:

```markdown
## rite Review Summary

### Overall Assessment: {Recommend Merge/Conditional/Block}

### Reviewer Consensus
| Reviewer | Assessment | Critical | High | Medium | Low |
|----------|------------|----------|------|--------|-----|
| Security | Pass | 0 | 0 | 1 | 2 |
| DevOps | Conditional | 0 | 1 | 0 | 0 |

### Cross-Validated Findings
[High-confidence issues found by multiple reviewers]

### All Findings
[Complete findings from all reviewers]

### Next Steps
[Recommended actions based on findings]
```

## Overall Assessment Criteria

| Assessment | Condition |
|------------|-----------|
| **Recommend Merge** | Critical: 0, High: 0 |
| **Conditional** | Critical: 0, High: 1-2 (all fixable) |
| **Block** | Critical: 1+ or High: 3+ |

## Emoji Usage Policy

Emojis are used only for the following visibility purposes:
- Unified report header (`📜 rite レビュー結果`)
- Work memory identifier (`📜 rite 作業メモリ`)
- Important warning display (`⚠️ 矛盾する指摘を検出`)

Individual skill file Findings output must not use emojis.
