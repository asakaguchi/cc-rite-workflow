# テスト仕様書ディレクトリ

rite workflow のプロンプトエンジニアリング成果物（commands/*.md）に対するテスト仕様書を格納するディレクトリ。

## 概要

rite workflow はコードベースではなく、Claude Code 用のプロンプト（Markdown ファイル）で構成されているため、従来のユニットテストやインテグレーションテストとは異なるテストアプローチを採用しています。

## テスト方式

### 1. ドキュメント整合性テスト

- GraphQL クエリの構文・構造確認
- CLI コマンドのパラメータ整合性確認
- エラーハンドリングの一貫性確認

### 2. 仕様テスト

- 期待される入出力の明確化
- エッジケースの網羅
- 設定値の整合性確認

### 3. 手動検証手順

- 実際のリポジトリで検証可能なコマンド例
- 期待される出力の明示

## テスト仕様書の命名規則

### コマンドセクションテスト

```
{command}-{section}.test.md
```

例: `<command>-<section>.test.md`（コマンドの特定 Phase / セクションを対象とするテストの命名形式）

### Issue ベーステスト

```
{feature}-{issue_number}.test.md
```

例:
- `skill-structure-358.test.md` - スキル構造改善（PDFガイド準拠）の検証テスト

### シェルスクリプトテスト

```
{script-name}.test.md
```

例（命名形式のみ。対応するシェルスクリプトと同名の `.test.md` を置く）:
- `<script-name>.test.md` — `<script-name>.sh` のテスト仕様書

対応するテストスクリプト（`*.test.sh`）は対象スクリプトと同階層の `tests/` ディレクトリに配置する。

## テスト仕様書の構成

```markdown
# テスト仕様書: {対象}

## 概要                         ← 必須
## 対象機能                     ← 必須
## テストケース                 ← 必須
### TC-001: {テストケース名}
### TC-002: ...
## 改善提案（発見された課題）    ← 任意
## 付録                         ← 任意
## テスト実行                   ← 自動テストがある場合
## テスト実行記録               ← 必須
## 変更履歴                     ← 必須
```

**注**: 「必須」マークのセクションは全テスト仕様書に含めてください。「任意」セクションはテスト内容に応じて省略可能です。「テスト実行」は自動テストスクリプトがある場合のみ記載します。必須/任意の明示が導入される前に作成された既存のテスト仕様書も必須セクションを含んでいます。

## テスト実行

テスト仕様書は手動で検証するか、CI/CD パイプラインで自動検証することを想定しています。

### 手動検証

1. テスト仕様書の「手動検証手順」セクションに記載されたコマンドを実行
2. 「期待される出力」と実際の出力を比較
3. 「テスト実行記録」に結果を記録

### 自動検証（将来構想）

- GraphQL クエリの構文チェック
- CLI コマンドのヘルプ出力との整合性確認
- 設定ファイルとの整合性確認
