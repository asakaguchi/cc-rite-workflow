# テスト仕様書: start.md 既存 PR 検出時の自動遷移

## 概要

`plugins/rite/commands/issue/start.md` の「再開時の動作」セクションにおける既存 PR 検出時の自動遷移ロジックに対するテスト仕様書。

## 対象機能

`/rite:issue:start {number}` を再度実行した際に、既存ブランチに対応する PR が存在する場合、PR の状態に応じて適切なコマンドに自動遷移する機能。

---

## テストケース

### TC-001: PR 検出コマンドの構文確認

**目的**: PR 検出に使用する `gh` コマンドが正しい構文であることを確認する。

**対象コマンド** (`start.md` 行 2993-2996):

```bash
gh pr list --head {branch_name} --json number,title,state,isDraft --jq '.[0]'
```

**検証項目**:

| # | 検証項目 | 期待値 | 結果 |
|---|---------|--------|------|
| 1 | `--head` オプション | ブランチ名でフィルタリング | - |
| 2 | `--json` で取得するフィールド | `number`, `title`, `state`, `isDraft` | - |
| 3 | `--jq` の出力 | 最初の 1 件のみ抽出 | - |
| 4 | PR が複数ある場合 | 最初の 1 件を使用 | - |

**手動検証手順**:

```bash
# 実際のリポジトリで検証
# 1. 既存の PR があるブランチで実行
gh pr list --head "feat/issue-123-example" --json number,title,state,isDraft --jq '.[0]'

# 2. PR がないブランチで実行（空出力を確認）
gh pr list --head "nonexistent-branch" --json number,title,state,isDraft --jq '.[0]'
```

**期待される出力（PR が存在する場合）**:

```json
{
  "isDraft": true,
  "number": 123,
  "state": "OPEN",
  "title": "feat: add new feature"
}
```

### TC-002: PR 状態に応じた自動遷移の確認

**目的**: PR の状態ごとに正しいコマンドに遷移することを確認する。

**対象セクション** (`start.md` 行 2999-3021):

**遷移ルール**:

| PR 状態 | 条件 | 遷移先 |
|--------|------|--------|
| OPEN + Draft | `state == "OPEN" && isDraft == true` | `rite:pr:review` |
| OPEN + Ready | `state == "OPEN" && isDraft == false` | `rite:pr:review` |
| MERGED | `state == "MERGED"` | `rite:pr:cleanup` |
| CLOSED | `state == "CLOSED"` | ユーザーに確認 |

**検証項目**:

| # | 検証項目 | 期待値 | 結果 |
|---|---------|--------|------|
| 1 | Draft PR の遷移先 | `rite:pr:review` | - |
| 2 | Ready PR の遷移先 | `rite:pr:review` | - |
| 3 | Merged PR の遷移先 | `rite:pr:cleanup` | - |
| 4 | Closed PR の遷移先 | ユーザー確認（`AskUserQuestion`） | - |

### TC-006: 検出タイミングの確認

**目的**: PR 検出が正しいタイミング（Phase 2.2 で既存ブランチを検出した直後）で行われることを確認する。

**対象セクション** (`start.md` 行 2991-2993):

**検証項目**:

| # | 検証項目 | 期待値 | 結果 |
|---|---------|--------|------|
| 1 | 検出タイミング | Phase 2.2 で既存ブランチを検出した直後 | - |
| 2 | 検出条件 | 既存ブランチに切り替えた場合のみ | - |
| 3 | 新規ブランチ作成時 | PR 検出はスキップ | - |

**フローチャート**:


| # | 検証項目 | 期待値 | 結果 |

## 改善提案

テストケース作成時に発見された仕様の課題:

### IP-001: 複数 PR 存在時の明示的な動作定義

**発見元**: TC-007

**観察**: 同一ブランチに複数の PR が存在するケースについて、仕様書に明示的な記載がない。

**現状**: `--jq '.[0]'` により暗黙的に最初の 1 件が選択される。

**提案**: 以下の注釈を追加することを検討:

```markdown
**注**: 同一ブランチに複数の PR が存在する場合、最も最近更新された PR が選択されます。
```

**優先度**: Low（実用上問題になるケースは稀）

---

### IP-002: PR 再オープン後の動作フロー

**発見元**: TC-004

**観察**: CLOSED 状態の PR を「再オープンして作業を続ける」を選択した後の具体的なフローが記載されていない。

**提案**: 以下のフローを追記することを検討:

```markdown
**「PR を再オープンして作業を続ける」選択時:**

1. `gh pr reopen {pr_number}` で PR を再オープン
2. `rite:pr:review` を呼び出す
```

**優先度**: Medium（ユーザーが混乱する可能性あり）
