### Fix Patterns (cycle 2)

**bash pitfall**: `if ! cmd; then rc=$?` は bash の既知バグで `!` 否定が rc を反転するため then 節内の `$?` は常に 0 を返す。error message の `rc=$rc` 値が実際のエラーコードでなく常に 0 で誤表示される。canonical pattern は `cmd; rc=$?; if [ "$rc" -ne 0 ]` の 2 文分割、または `if cmd; then :; else rc=$?; fi` の else 節形式。Phase 2.2 / 2.3 が既に canonical pattern を採用していたのに、Phase 6.0 新設時に逆 pattern を採用してしまい既知バグを再導入した事例。

**LLM state communication via 4 値 enum**: bash 変数は Bash tool 呼び出し境界を超えて失われるため、後段 Phase で LLM が参照する状態を bash block 内で set する場合、stdout に `key=value` 形式で明示出力する必要がある。2 値 enum (true/false) では legitimate absence と IO error の区別が失われて false positive 警告が機能しない。4 値 enum (unknown/true/absent/io_error) で状態空間を明示し、Phase 9.1 完了レポート template 側で値ごとの表示条件を定義する。

**triple cross-validated finding**: 3 reviewer が独立に同じ dead flag 問題 (log_read_ok が stdout 未出力) を検出。1 回の review で同じ設計欠陥を複数 reviewer が独立発見した場合、問題の存在確率が極めて高く、severity boost の候補になる (rite では HIGH 据え置きで実害影響の程度で判定)。
