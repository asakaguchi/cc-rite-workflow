### Fix Patterns

PR #1155 cycle 1 fix で以下 3 種類の silent regression を回収した経験則:

1. **mktemp 失敗時の silent fallback 格下げ regression**: `var=$(mktemp 2>/dev/null) || var=""` 形式は Pattern 3 規範 (`bash-trap-patterns.md`) の loud WARNING 3 行を骨抜きにする。同一 PR 内で同型の mktemp 呼び出しがある場合、すべての site で対称化されているかを grep で確認する習慣を持つ。本 PR では 9 件の mktemp 呼び出しのうち 2 件 (ingest.md L248 cat_err / L535 _reset_err) のみが silent fallback になっており、対称化検査で検出可能だった。

2. **YAML パース helper の DRY 化と semantic preservation の trade-off**: 重複パース処理を helper 化 (extract_yaml_key) する際、key 別の subtle な差異 (lowercase 変換の有無) を **silent に抹消する** behavior change を内包しやすい。修正前 develop 版は wiki_enabled のみ lowercase 適用、wiki_branch / branch_strategy は大文字保持の挙動を持っていたが、helper 化で 3 key 一律 lowercase になり mixed case branch 名の silent corrupt を引き起こす。helper 抽出前後で「全 key の挙動が完全に等価か」を確認するルールを抽象化すべき。

3. **「フラット化」PR スコープの誤適用 (歴史注釈 vs 現役の同期保守情報)**: 「declarative defense layer の物理排除」目標で blockquote / 歴史注釈を削除する際、(a) 削除対象は git blame で追える歴史注釈、(b) 削除対象でないものは現役の同期保守情報 (分散実装一覧、Pattern 規範の WARNING emit、Reference: link) であることを分類する必要がある。混同すると次 cycle で reviewer が新規 finding を出して fix loop が永久化する。SoT 化 (`Single Source of Truth` 構造への集約) は (b) の喪失を防ぐ canonical 対処法。

### 一般化された経験則

- **対称性違反検出 = silent regression 検出**: 同型 site の 1 つだけが「軽量化」されている状態は概ね silent regression。grep で `mktemp ... || var=""` パターンを列挙し、Pattern 3 規範に従う block と非対称な block を機械的に特定可能。
- **helper 化と挙動差分**: DRY 化リファクタは挙動差分を生む潜在経路。新 helper 導入時は「helper 化前後で全呼び出し site の挙動が bit-identical」を verify する unit test 相当の作業が必要 (本 PR ではこれが欠落していた)。
- **「フラット化」と「機能削除」の分離**: cleanup.md スタイル本格フラット化は文書 layout の問題で、機能 statement (WARNING emit / 同期保守情報) は対象外。スコープ判定 checklist として「git blame で追える歴史か / 現在の reader が判断に必要か」を二軸で評価する。
