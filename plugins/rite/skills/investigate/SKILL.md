# /rite:investigate

## Contract
**Input**: 調査対象の説明（必須）
**Output**: 検証済みの調査結果テーブル + 最終報告

ソースコードを正確に調査し、検証済みの結果を報告する。grep で場所を特定し、Read で実際のコードを確認し、Codex（または代替検証）でクロスチェックする3段階検証を含む7フェーズのプロセスにより、推測による誤報告を防ぐ。

> **簡易版**: 他フェーズでコード調査が必要な場合は [investigation-protocol.md](../../references/investigation-protocol.md) を参照。

---

## 背景

AI エージェントによるコード調査では、grep 結果の行番号だけに頼ってコード構造を推測してしまう問題が繰り返し発生した。特に複数行にまたがる構造（関数呼び出し、ハッシュ定義、条件式チェーン等）や、大きなファイル（数百行超）で顕著。この問題を防ぐため、「grep → Read → クロスチェック」の3段階チェックを標準化する。

## 引数

```
/rite:investigate <調査対象の説明>
```

例:
- `/rite:investigate audit_log の全呼び出しで使われている target_type と op_type`
- `/rite:investigate useEffect の依存配列に含まれる変数の一覧`
- `/rite:investigate API エンドポイントとそのハンドラー関数の対応`
- `/rite:investigate rite-config.yml の全キーとそのデフォルト値`
- `/rite:investigate if/elsif 分岐で使われている条件式パターン`
- `/rite:investigate メソッドチェーンの終端で呼ばれているメソッド一覧`

## 調査プロセス

### Phase 1: スコープの明確化

ユーザーの指示から以下を特定する:

| 要素 | 説明 | 例 |
|------|------|-----|
| **対象** | 何を調べるか | `audit_log` の全呼び出し |
| **観点** | どの情報を抽出するか | 第1引数（target_type）と第2引数（op_type） |
| **範囲** | どのディレクトリ/ファイルを調べるか | `app/controllers/` 配下の `.rb` ファイル |

不明確な場合は AskUserQuestion で確認する:

```
調査内容を確認させてください。

オプション:
- リポジトリ全体を調査
- 特定のディレクトリに限定（パスを入力）
- 特定のファイルに限定（パスを入力）
```

**観点が不明確な場合**:

```
何を抽出・確認しますか？

オプション:
- 出現箇所の一覧（ファイル:行番号）
- 引数・パラメータの値
- 構造の定義内容（キー、条件、メソッド等）
- 詳細を説明する
```

### Phase 2: 検索（Grep）

Grep ツールで対象パターンを検索し、該当ファイルと行番号を特定する。

```
Grep: パターンを検索 → ファイル一覧を取得
```

この段階では場所の特定のみ。値の読み取りは次の Phase で行う。

#### 2段階検索（複数行にまたがる構造対策）

コード構造の開始と検索対象が別の行にある場合、単一パターンの grep では見逃す。以下の2段階で検索する:

**Step 1**: 構造の開始パターンで検索して全出現箇所を特定
```
例: Grep で "audit_log" を検索 → 全出現箇所の行番号を取得
例: Grep で "class.*Controller" を検索 → 全クラス定義の行番号を取得
例: Grep で "config\[" を検索 → 全ハッシュアクセスの行番号を取得
```

**Step 2**: 検索対象のキーワードで検索してファイルを絞り込み
```
例: Grep で "bigquery" を検索 → 該当ファイルを特定
例: Grep で "enabled" を検索 → 該当設定項目を特定
```

**照合**: Step 1 と Step 2 の結果を照合する。Step 2 でヒットしたファイルに対して、Step 1 の全出現箇所を Phase 3 で Read 検証する。Step 2 のキーワードが構造の一部として使われているかどうかは、Read で確認するまで判断しない。

この2段階検索により、以下のような複数行にまたがるケースも漏れなく検出できる:
- Ruby 関数呼び出し: `audit_log(` が行376、`'bigquery.query'` が行377
- Ruby ハッシュ定義: `config = {` が行10、`enabled: true` が行15
- Python デコレータ: `@app.route(` が行20、`def handler(` が行21
- JavaScript/TypeScript: `useState(` が行15、初期値 `{ name: '' }` が行16
- 条件式: `if` が行50、`elsif`/`else if` が行55、`else` が行60
- メソッドチェーン: `.where(` が行100、`.order(` が行101、`.limit(` が行102
- YAML/設定ファイル: キー `investigate:` が行133、ネストされた `enabled:` が行135

### Phase 3: 検証（Read）

**これが最も重要なフェーズ。** Grep で特定した各箇所を Read ツールで実際に確認する。

#### 必須ルール

1. **構造の開始から終了まで読む**: コード構造が複数行にまたがる場合、1行目だけでは内容が見えない。最低でも該当行の前後5行を含めて読む。構造が5行を超える場合は構造の終了行まで読む
2. **推測で値を補完しない**: Read で確認できない値は「未確認」と報告する。「たぶん create だろう」のような推測は禁止
3. **変数は定義元を追跡する**: 値が変数の場合、その変数の定義・代入箇所も Read で確認する
4. **大きなファイルは offset/limit で分割読み**: 数百行超のファイルを全部読む必要はない。該当行の周辺だけを正確に読む
5. **全件処理を保証する**: Phase 2 の grep ヒット件数と Phase 3 で Read 検証した件数を照合する。一致しない場合は未検証の箇所を特定して追加で Read する

#### 完全性チェック（Phase 3 の最後に必ず実行）

Phase 2 の grep で N 件のヒットがあった場合、Phase 3 で N 件すべてを Read で検証したことを確認する。

```
grep ヒット数: N 件
Read 検証数:   M 件
未検証:       N - M 件 → 追加で Read が必要
```

大きなファイルで grep ヒットが離れた行にある場合（例: 行376 と行491）、ファイルの一部だけ読んで「全部確認した」と判断しがち。grep 結果の全行番号リストを作り、各行番号を Read でチェックオフしていく。

#### 出力形式

各ファイルごとに以下のテーブルで記録する:

```markdown
### {ファイルパス}

| 行番号 | {観点1} | {観点2} | 実際のコード |
|--------|---------|---------|-------------|
| 123 | `値1` | `値2` | `構造名('値1', '値2', ...)` |
```

「実際のコード」列には、Read で確認した実際のソースコードの該当部分を引用する。これが証拠となる。

### Phase 4: 動的パターンの追跡（該当する場合）

文字列補間（`"#{variable}"`）、条件分岐による値の変化、ラッパー関数/メソッドの中身など、静的な grep だけでは追えないパターンがある場合:

1. 変数の定義元を Read で確認
2. 条件分岐の全パターンを列挙
3. ラッパー関数/メソッドの内部実装を確認し、値を変換していないか検証
4. メソッドチェーンの中間結果を追跡

### Phase 5: 結果の集約

重複を排除した最終一覧を作成する。

```markdown
## 全 {調査対象} 一覧（重複なし）

| # | {観点1} | {観点2} | 検証ソース |
|---|---------|---------|-----------|
| 1 | `値1` | `値2` | ファイル名:行番号 |
```

「検証ソース」列は、Phase 3 で Read により確認したファイルと行番号。この列があることで、後続のクロスチェックや手動検証が可能になる。

### Phase 6: クロスチェック

> **設定読み取り**: `rite-config.yml` の `investigate.codex_review.enabled` を確認する。未設定の場合はデフォルト `true` として扱う。

#### Phase 6a: Codex クロスチェック（`enabled: true` かつ Codex MCP 接続時）

調査結果を Codex に渡してクロスチェックする。Codex には以下を依頼する:

1. 各項目の値が検証ソースの行番号で正しいか確認
2. 調査範囲内に見落としがないか再チェック
3. 検出結果のサマリー

Codex に渡すプロンプトには、Phase 5 の結果テーブルと検証ソースの行番号を含める。Codex が実際のソースコードを読んで検証できるようにする。

Codex MCP（`mcp__codex__codex`）が利用できない環境では、Phase 6b にフォールバックする。

#### Phase 6b: 代替検証（`enabled: false` または Codex MCP 未接続時）

Claude 自身が Phase 5 の結果を再検証する:

1. **サンプル再検証**: 結果テーブルからファイルが偏らないよう分散させて最大5件を選び、Read ツールで再度該当行を確認する
2. **網羅性チェック**: Phase 2 の grep パターンを変形（同義語、別の記法）して再検索し、見落としがないか確認する
3. **整合性チェック**: 結果テーブル内の値同士に矛盾がないか検証する

代替検証で不一致が見つかった場合は、Phase 3 に戻って該当箇所を再検証する。

### Phase 7: 最終報告

クロスチェックの結果を反映した最終報告を出力する。

```markdown
## 調査結果

### サマリー
- 調査対象: {対象}
- 検出数: {件数}
- Codex クロスチェック: {通過 / 修正あり / 未実施（代替検証） / 未実施（MCP 未接続）}

### 結果テーブル
{Phase 5 のテーブル（クロスチェックの修正を反映）}

### クロスチェック結果
{修正があった場合の詳細}
```

## よくある失敗パターンと対策

| 失敗パターン | 原因 | 対策 |
|-------------|------|------|
| コード構造の値が間違っている | grep の1行目だけ見て推測した | Read で構造の開始〜終了まで確認 |
| 一部の出現箇所が漏れている | ラッパー関数/メソッド経由の呼び出しを見逃した | ラッパーの定義を確認し、内部の呼び出しも追跡 |
| 変数の値が間違っている | 変数の定義元を確認せず推測した | 変数の代入箇所を Read で確認 |
| 大きなファイルで部分的にしか確認していない | ファイル全体を読もうとして途中で切れた | offset/limit で該当行の周辺だけを正確に読む |
| 複数行にまたがる構造が漏れる | 単一行 grep では複数行の構造にマッチしない | 2段階検索（Phase 2）で構造開始とキーワードを分離して検索 |
| ハッシュのキーと値の対応を誤認 | インデントだけで対応を推測した | Read でハッシュの開始〜終了を確認し、ネスト構造を把握 |
| 条件分岐の全パターンを見逃す | 最初の `if` だけ確認して `elsif`/`else` を見逃した | 条件式の開始〜終了（`end`/`}`/同インデント行）まで Read |
| メソッドチェーンの途中結果を見誤る | チェーンの最初と最後だけ確認した | チェーン全体を Read で確認し、各ステップの戻り値を追跡 |
