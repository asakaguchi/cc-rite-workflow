### 指摘内容

**F-01 (error-handling, MEDIUM)**: `plugins/rite/hooks/session-start.sh:392` の `.gitignore` 生成行 `printf '*\n' > "$_reap_log_dir/.gitignore" 2>/dev/null || true` は、bash のリダイレクト評価順序（`>` が先、`2>/dev/null` が後）により、ファイルオープン自体が失敗した場合（読み取り専用ディレクトリ、パス衝突等）に bash 自身が出す診断メッセージ（例:「許可がありません」）が `2>/dev/null` で抑止されず stderr に漏れる。

直前の行（line 391）にある既存パターン `{ : > "$_reap_log_dir/pr-cycle-cleanup.log"; } 2>/dev/null` は brace group で囲むことでこの問題を正しく回避しているが、line 392 はこのパターンに従っていなかった。

### 教訓（経験則候補）

- **同一ファイル内でリダイレクト抑止パターンが不統一だと再発する**: line 391 で確立した「brace group で囲んでから `2>/dev/null` を外側に付ける」パターンが、数行後の line 392（cycle 4 で新規追加されたコード）で再現されなかった。これは「bash の `cmd > file 2>/dev/null` は `>` が先に評価される」という落とし穴（他の PR/cycle でも繰り返し指摘されている汎用パターン）が、同一ファイル内の既存の正しい実装を横目に見ていても、新規追加コードでは再度踏まれうることを示す。新しいリダイレクトを書くときは、同一ファイル内の既存の正しいパターンをテンプレートとしてコピーするのが安全。
- **実機検証で severity 判定の確度を上げる**: 「診断メッセージが stderr に漏れる」という主張は、read-only ディレクトリでの再現実験（`{ printf ... > file; } 2>/dev/null` は 0 バイト、brace group なしは 93 バイト）で独立検証してから finding として確定させた。これにより false-positive リスクを排除しつつ、修正の必要性を客観的根拠付きで判断できた。

### 修正内容

`[ -f "$_reap_log_dir/.gitignore" ] || { printf '*\n' > "$_reap_log_dir/.gitignore"; } 2>/dev/null || true` に統一（line 391 と同一パターン）。回帰テスト TC-1968-08（`.gitignore` パスをディレクトリで衝突させて書込み失敗を再現）を追加し、mutation testing（修正前のコードに戻すと red になることを確認）で teeth を検証済み。
