## 修正実行 (PR #1969, cycle 5)

### 対応した指摘（1件）

**error-handling (MEDIUM)**: `plugins/rite/hooks/session-start.sh:392` の `.gitignore` 生成行がリダイレクト評価順序の問題で書込み失敗時に bash 診断メッセージを stderr に漏らす。

### 対応内容

修正前:
```bash
[ -f "$_reap_log_dir/.gitignore" ] || printf '*\n' > "$_reap_log_dir/.gitignore" 2>/dev/null || true
```

修正後（line 391 の brace group パターンに統一）:
```bash
[ -f "$_reap_log_dir/.gitignore" ] || { printf '*\n' > "$_reap_log_dir/.gitignore"; } 2>/dev/null || true
```

回帰テスト TC-1968-08 を追加（`.gitignore` パスをディレクトリで衝突させて書込み失敗を再現し、bash の `session-start.sh: 行 N:` 診断メッセージが stderr に出ないことを検証）。mutation testing（修正前のコードに一時的に戻す）で TC-1968-08 が red になることを確認済み — teeth あり。72 tests pass。

### 教訓（経験則候補）

- **`cmd > file 2>/dev/null` パターンは brace group で囲まないと open 失敗時の診断メッセージが漏れる**: bash はリダイレクトを左から右に評価するため、`>` (ファイルオープン) が先、`2>/dev/null` が後になる。ファイルオープン自体が失敗すると、そのタイミングでは `2>/dev/null` がまだ適用されておらず bash 自身の診断メッセージ（`行 N: <path>: 許可がありません` 等）が抑止されない。正しくは `{ cmd > file; } 2>/dev/null` のように brace group で括り、group 全体の stderr をリダイレクトする。
- **同一 PR 内で確立したパターンが数行後の新規コードで再現されないことがある**: このバグは cycle 4 で新規追加された行（line 392）で発生し、直前の行（line 391、cycle 1 由来）は同じ落とし穴を正しく回避していた。新しいリダイレクトを書く際は、同一ファイル内の直近の正しい実装をテンプレートとしてコピーするのが安全。
- **severity 判定前の実機検証で確度を上げる**: read-only ディレクトリでの再現実験により、修正前パターンは stderr に 93 バイト漏らし、修正後は 0 バイトであることを独立に確認してから finding を fix-needed として確定させた。
