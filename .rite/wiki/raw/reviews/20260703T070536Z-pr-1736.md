### Finding Patterns
- セキュリティ境界に timeout を追加する際、PreToolUse hook の timeout は fail-open のため、既存の super-linear 正規化処理と組み合わさると timeout→fail-open bypass を実用化しうる（timeout 値を下げると攻撃に必要な入力サイズも下がる）。timeout を安全境界の上限に当てにせず、評価コストを timeout 到達不能な範囲に抑える（oversized 入力は正規化前に fail-closed deny）。
- テスト専用の fault-injection を env var 分岐で本番セキュリティ hook に同梱する場合、fail-OPEN 方向（allow-all）に反転させる分岐は settings.json env 経由の silent バックドアになりうる。fail-closed 側（self-restrictive=deny のみ）の injection に限定する。
- 外部ツール（Claude Code hook）の timeout/fail 方向の主張は fact-check（公式 docs）で検証してから finding を確定する。reviewer 間の「trigger 経路なし」判断は解析スコープ（無限ループ vs super-linear コスト）の差で分かれることがある。
