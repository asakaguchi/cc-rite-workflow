### Lessons (anti-patterns reinforced — cycle 2 specific)

- **CHANGELOG `removed in #N` fact-check protocol** (新規): When adding CHANGELOG entries for `removed in #N` deletion-type changes, run `ls -d {path}` or `git show {sha}:{path}` to verify the claim against actual file system state. The CHANGELOG itself should not include unverified "remain on disk" claims when the actual `ls` returns ENOENT. This PR's cycle 1 reviewer caught a similar but smaller drift (observed_likelihood_gate vs separate_issue_creation), but missed the i18n/ja factual error because they trusted the CHANGELOG text without `ls` verification.

- **Asymmetric Fix Transcription — `# wave as unit of consistency`** (cycle 2 confirmation): When CHANGELOG references `# wave` (e.g., #1118) that retired multiple features in one commit, the SAME PR must complete the DEPRECATED-ification for ALL features in that wave. cycle 1 caught 3 keys (observed_likelihood_gate, fail_fast_first, severity_gating) but missed project.type, which was the same #1118 wave. The fix is to grep CHANGELOG entries for `# N` references, then `git show <sha>` to enumerate all touched files, and apply DEPRECATED-ification per file. The wave is the unit of consistency, not the individual key.

- **Multi-PR drift surfaced by docs review** (cycle 2 root cause): The actual root cause is that the original deletion PRs (#1117 / #1118) didn't include CONFIGURATION.md / style-guide drift fix as part of their acceptance criteria. The drift only surfaced 2 release cycles later during this v0.5.0 docs review. Lesson: future deletion PRs should include `grep -rn '<retired-key>' docs/` verification as a mandatory step before merge, not deferring to release-time docs review.
