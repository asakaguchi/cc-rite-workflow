### Fix Patterns

mkdir 成功だけで書き込み可否を判定する非対称フォールバックの是正パターン: `if mkdir -p DIR 2>/dev/null; then <実際に書く> else <破棄> fi` は、DIR が既存の読み取り専用ディレクトリの場合でも mkdir -p が rc=0 を返すため、後続のファイルオープン失敗をフォールバックできない。修正は mkdir 直後に `{ : > FILE; } 2>/dev/null` で truncate を試み、書き込み可否まで if 条件に含める。truncate 自体が「毎回上書き」の役割を兼ねるため、実際の処理は `>>` (append) に切り替える。

テストフィクスチャ選択の教訓: 対象コマンドが git チェックで早期 exit するツール（本ケースは pr-cycle-cleanup.sh）を非 git ディレクトリでテストすると、実際の完走経路（stdout 出力等）を検証できない false positive を生む。git 実フィクスチャ + TMPDIR 隔離（対象コマンドが tmp 配下を sweep する場合、開発機の実 TMPDIR を汚染しないため）への統一が必要。mutation testing（意図的にバグを再導入してテストが red になるか確認）で teeth の有無を検証する習慣が有効だった。
